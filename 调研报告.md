**关键结论**: BBDown 是**两个工程**:
- `BBDown` (CLI 壳) — 负责参数解析、文件 IO、ffmpeg 调用、HTTP server
- `BBDown.Core` (协议库) — 负责抓取、解析、API 调用

**Rust 重写时这个分层必须保留**—— `core` crate 应该是 `no_std` 友好的纯异步库（不依赖 ffmpeg 路径），`cli` crate 是壳。

### 1.2 核心架构 (4 个支柱)

#### ① 全局状态 (Config.cs)

```csharp
public static class Config {
    public static string COOKIE = "", TOKEN = "", COOKIE_TV = "";
    public static string HOST, API, EPHOST, BANGUMI_API, SEARCH_API, ...;
    public static bool ALLOW_WEB, ALLOW_APP, ALLOW_HTML5;
    public static int Quality, AudioQuality, ThreadCount, EncodingPriority, DfnPriority;
    public static string FilePattern, ConfigPath, Aria2cArgs;
    public static bool DownloadDanmaku, DownloadSubtitle, DownloadCover, DownloadChapter;
    public static bool IsLogin, IsLoginTV, UseAria2c, Interactive;
    public static Logger log = new();
    public static string Uuid, Cookie_Buvid, Cookie_Buvid3, Cookie_Buvid4, ...;
}
```

**Rust 对应**: `Arc<RwLock<AppConfig>>` 或 `OnceCell`，不要用 `static mut`。所有 `BUVID/UUID` 等 cookie 应在 `login` crate 启动时主动从 web 拉取并存到 `keyring`。

#### ② Fetcher 模式 (IFetcher.cs)

```csharp
public interface IFetcher {
    public Task<VInfo> FetchAsync(string id);
}
```

**返回 `VInfo` 包含**:
- 基础字段: `Title / Desc / Pic / PubDate`
- 多 P: `List<Page>`（每 Page 含 `Cid/PageIndex/Dfn/Dura/Title`）
- 单视频: `Aid/Bvid/Cids/PartNames`
- 互动: `List<ViewPoint>`
- 特殊类型: `IsBangumi / IsCheese / IsSteinGate`
- **鉴权要求**: `LoginState` 枚举（`None / Web / Login / Token`）—— **Fetcher 自己声明需要什么登录级别**

**Factory (FetcherFactory.cs)** 用正则分流:
- `BV/av/upid` → `NormalInfoFetcher`
- `ss/ep/md` → `BangumiInfoFetcher`
- `intl.xxx` 番剧 → `IntlBangumiInfoFetcher`
- `/cheese/` → `CheeseInfoFetcher`
- `/favlist/` → `FavListFetcher`
- `/medialist/` → `MediaListFetcher`
- `/series/` → `SeriesListFetcher`
- `/space.` → `SpaceVideoFetcher`

**Rust 对应**:
```rust
#[async_trait]
pub trait Fetcher: Send + Sync {
    async fn fetch(&self, id: &str) -> Result<VInfo>;
    fn login_requirement(&self) -> LoginState { LoginState::None }  // 默认方法
}
```

#### ③ Parser 模式 (Parser.cs — 4 种 playurl)

B 站获取视频流有 **5 种接口**，BBDown 全部实现:

| 模式 | 接口 | 鉴权 | 限制 | 文件大小 |
|---|---|---|---|---|
| HTML5 | `/x/playurl` | 无/cookie | 单 FLV | BBDownUtil.GetWebSource fallback |
| Web | `/x/web-interface/view` + `/playurl` | SESSDATA | 非登录最大 480P | Parser.GetPlayJsonAsync |
| Web 番剧 | `/pgc/view/web/season` + `/playurl` | SESSDATA | 需大会员 | 同上 |
| TV | `/pgc/view/web/season` + `/pgc/player/web/playurl` | TV token | 部分番剧 | 同上 |
| APP | `playview` 端点 (gRPC, 需 protobuf header) | access_token | 全部质量 | AppHelper.PlayView (582 行) |

**关键算法 (Wbi 签名)**:
```csharp
// 1. 从 https://api.bilibili.com/x/web-interface/nav 拿 img_url 和 sub_url
// 2. 取 query string "?r=xxx" 后的 key，再 XOR 字符表
// 3. mixin_key = 对前 32 位做 64-round 重排 (table[0..32])
// 4. w_rid = MD5(param + mixin_key)  ← 这就是 WBI 签名
```

**视频流提取 (`ExtractTracksAsync`)**:
- 解析 `accept_quality[]` (8K/HDR/杜比/4K/1080P60/1080P+/1080P/720P/480P/360P/144P)
- 杜比视界检测: `qn=125` + `has_dolby=true` + `bluerayElectric/qhdHDRVivid` 标志
- 杜比全景声: `flac` 容器 + `dolbyAtmos` 标志
- 帧率: `frameRate == 60` (高帧率)
- 编码: `codecid` → AVC(7) / HEVC(12) / AV1(13)
- DASH vs FLV: 有 `dash` 字段 → 视频 m4s / 音频 m4s；否则 `durl` 字段 → 单 FLV

#### ④ AppHelper 模式 (APP gRPC)

BBDown 走的是 B 站 **手机端 APP 的 gRPC 协议**:
1. 构造 6 段 protobuf header: `device / fawkesreq / locale / metadata / network / restriction`
2. body 用 `dmviewreq` 或 `playviewreq`
3. **gzip 压缩 body**
4. 走 `https://grpc.biliapi.net/bilibili.app.playurl.v1.PlayURL/PlayView`
5. 解析 `playviewreply`（可能再次 gzip）

这是 BBDown **最高质量 (8K/HDR/杜比) 唯一能拿到的接口**——TV 和 Web 都有 1080P 限制。

### 1.3 登录流程 (BBDownLoginUtil.cs)

**WEB 登录（PC 端 SESSDATA cookie）**:
```
1. 调 /x/passport-login/web/qrcode/generate
   → 返回 qrcode_key + qrcode_url
2. ConsoleQRCode 把 url 渲染成 ASCII 二维码
3. 轮询 /x/passport-login/web/qrcode/poll?qrcode_key=xxx
   状态: 86101 (未扫) / 86090 (已扫未确认) / 0 (成功)
4. 成功 → 从 Set-Cookie 提取 SESSDATA / bili_jct / DedeUserID / DedeUserID__ckMd5
5. 写到 BBDown.config (明文 JSON)
```

**TV 登录**:
```
调 /x/passport-login/tv/qrcode/auth_code + /x/passport-login/tv/qrcode/poll
→ 拿到 access_token + refresh_token + mid
→ 用于 TV 端 gRPC playview 鉴权
```

**关键观察**:
- BBDown 把 SESSDATA **明文写入 BBDown.config**——有泄露风险
- 没从 `https://www.bilibili.com/` 进入拿 `buvid3/buvid4/b_nut` —— 高质量接口会被风控
- TV token 没做 refresh 机制，30 天过期后只能重扫

### 1.4 下载流程 (Program.cs DownloadPageAsync)

单 page 下载的完整流程:

```
FetchAsync(id)
  ↓ VInfo
对每个 Page 循环:
  Parser.GetPlayJsonAsync(cid, bvid, aid, ...)
    → 4-mode playurl (HTML5 / Web / Web-bangumi / TV / APP)
  Parser.ExtractTracksAsync
    → 解析出 List<Video> / List<Audio> / List<Subtitle>
  按用户选项过滤 tracks
    → Quality / Codec / AudioQuality / EncodingPriority / DfnPriority
  下载 cover (可选)
  下载 danmaku (可选) → ASS
  下载 subtitles (可选) → SRT/ASS via SubUtil
  选最佳 video track (含杜比视界 mp4box 分支)
  选最佳 audio track
  视频: MultiThreadDownloadFileAsync / Aria2c
  音频: 同上
  混流: BBDownMuxer (ffmpeg / mp4box)
  元数据写入: -metadata title/desc/artist (用 ffprobe 提取后回写)
  删除临时片段
```

**多线程下载 (`MultiThreadDownloadFileAsync`)**:
```csharp
// 1. HEAD 请求拿 Content-Length
// 2. 计算每段字节 range: chunk = length / threadCount
// 3. 每线程一个 HTTP Range 请求
// 4. 用 Channel<int> 累计已下载字节
// 5. 所有线程完成后，文件流式拼接
```

**Aria2c 模式**:
- 调用本地 aria2c 守护进程（默认 6800 端口）
- `aria2.addUri` RPC 调用，传入 header 字符串
- 轮询 `aria2.tellStatus(gid)` 拿进度
- BBDown 主线程通过 `BBDown.config` 里 `Aria2cArgs` 透传额外参数

### 1.5 混流 (BBDownMuxer.cs)

**3 种场景**:

| 输入 | 工具 | 命令 |
|---|---|---|
| DASH m4s 视频 + m4s 音频 | ffmpeg | `ffmpeg -i v.m4s -i a.m4s -c copy -strict experimental out.mp4` |
| FLV 单文件 | ffmpeg | `ffmpeg -i in.flv -c copy out.flv`（多段要 concat）|
| **杜比视界 (DV) 视频** | **mp4box** 或 ffmpeg 5.0+ | `mp4box -add v.mp4 -add a.m4s out.mp4` |

**关键检测**:
- `is_dolby_vision` 标志 + 视频 `codecs` 字段包含 `dvh1/dvhe` 时走 DV 分支
- DV 走 ffmpeg 需要 ffmpeg 5.0+ 编译并开启 `--enable-libdav1d / --enable-libx265`
- 临时目录: `C:\Users\<user>\AppData\Local\Temp\`

**字幕/章节/封面的 mux 标志**:
- `-map 0:v -map 0:a -map 0:s? -c copy -c:s mov_text`
- `-metadata:s:v:0 language=zh-Hans`

### 1.6 弹幕 (DanmakuUtil.cs)

**输入 3 种**:
1. 旧 XML: `https://comment.bilibili.com/{cid}.xml`
2. Web API: `/x/v2/dm/web/seg.so?type=1&oid={cid}&segment_index=1..N`（protobuf 格式，需 zlib inflate）
3. 视频预览弹幕: `/x/v2/dm/view?type=1&oid={cid}`（新 protobuf 格式）

**输出**: 单一 `.ass` 文件
- 滚动/顶部/底部三种样式
- 颜色（用户等级/会员）
- 字号按播放时间动态调整
- 边框宽度基于屏幕比例

**Rust 重写要点**: protobuf 弹幕是 zlib 压缩的——`flate2::read::ZlibDecoder` + `prost` 解 `DmwV2.DanmakuElem`。

### 1.7 字幕 (SubUtil.cs - 5 套接口)

BBDown 实现了一套回退链:

```
1. /x/player/wbi/v2?cid=xxx           # 主用，需 WBI 签名
2. /x/player/v2?cid=xxx                # 旧 fallback
3. /x/player.so?id=cid                 # JSON
4. /x/web-interface/view?bvid=xxx      # 视频元信息里的 subtitle
5. /x/v2/dm/view?type=1&oid=cid        # 含字幕元数据
```

每条字幕支持 3 种输入格式: ASS / JSON / SRT，统一输出 SRT（用 `SubUtil.ConvertSubToSrt`）或保持原格式（用户指定）。

监听 **58682 端口**，ASP.NET Core Minimal API:

| 端点 | 方法 | 用途 |

### 1.10 BBDown 关键洞察 (供 Rust 设计参考)

✅ **做得好的**:
- **清晰分层**: CLI 壳 + Core 协议库
- **Fetcher 模式** + 工厂路由——多内容类型扩展容易
- **5 种 playurl 全支持**——保证高质量可达
- **WBI 签名实现**完整
- **杜比视界 / 全景声** 完整支持
- **多线程 / aria2c** 双下载器可切换
- **HTTP server 模式** 给 GUI 端用

⚠️ **可改进的**:
- **明文存 cookie** —— 改用系统 keychain (keyring)
- **buvid 没从主站取** —— 容易被风控
- **TV token 不自动 refresh** —— 30 天过期
- **没有连接复用** —— `HttpClient` 全局单例但 cookie 头没重置
- **单线程 await 混流** —— 大文件慢
- **APP gRPC 没实现续传**
- **DASH segment 解析死板** —— 只支持 startWith
- **DASH 多分片不能并发**（用单连接）
- **没实现 aria2c 的事件流** —— 是 polling
- **API server 状态全在 Config static** —— 竞态

### 2.3 关键算法实现 (Rust 改写可借鉴)

#### ① BV/AV 转换 (`types.py`)

```python
XOR_CODE = 23442827791579
MASK_CODE = 2251799813685247
MAX_AID = 1 << 51
ALPHABET = "FcwAPNKTMug3GV5Lj7EJnHpWsx4tb8haYeviqBz6rkCy12mUSDQX9RdoZf"
ENCODE_MAP = (8, 7, 0, 5, 1, 3, 2, 4, 6)

def bv2av(bv: str) -> int:
    bv = bv[2:]  # strip "BV"
    tmp = 0
    for ch, i in zip(bv, ENCODE_MAP):
        tmp = tmp * 58 + ALPHABET.index(ch)
    return (tmp - ADD_CODE) ^ XOR_CODE
```

> **Rust 版可直接搬**：纯整数运算，无 GC 压力。

#### ② `fnval=4048` 一次拿全画质

#### ③ 多 URL 并发探测 + Mirror 选最优

```python
async def first_successful_with_check(
    urls: list[UrlMeta],
    headers: dict,
    semaphore: asyncio.Semaphore,
) -> tuple[UrlMeta, int]:
    """HEAD 探测每个 URL, 选第一个 200/206"""
    async def head_with_check(url_meta: UrlMeta) -> tuple[UrlMeta, int] | None:
        async with semaphore:
            resp = await client.head(url_meta.url, headers=headers)
            if resp.status_code in (200, 206):
                return (url_meta, resp.status_code)
    # 并发探测, 第一个成功的胜出
    results = await asyncio.gather(
        *(head_with_check(u) for u in urls), return_exceptions=False
    )
    return next(r for r in results if r is not None)
```

**BBDown 缺这个**——它只用一个 URL，挂了重试。

> **Rust 改写**: 用 `tokio::sync::Semaphore` + `tokio::select!` + `FuturesUnordered` 同样实现。

```rust
// 并行转换多个输入文件
pub fn convert_to_ass(input: &Input) -> Result<String> {
    let mut output = String::new();
    write_head(&mut output, &config)?;
    
    let comments = match input {
        Input::Xml(s) => parse_xml(s)?,
        Input::Protobuf(b) => parse_protobuf(b)?,
    };
    
    // ⭐ 用 rayon 并行排序/过滤
    let filtered: Vec<_> = comments
        .into_par_iter()                       // 并行!
        .filter(|c| !block_options.is_blocked(c))
        .collect();
    
    let mut sorted = filtered;
    sorted.par_sort_unstable_by(|a, b| {        // 并行排序
        (a.timeline, a.timestamp, a.no, ...)
            .cmp(&(b.timeline, b.timestamp, b.no, ...))
    });
    
    // 顺序写入 ASS (row 冲突需要顺序)
    let mut row = Row::new(...);
    for c in sorted {
        if let Some(r) = row.find_alternative_row(&c) {
            write_normal_comment(&mut output, &c, r)?;
        }
    }
    Ok(output)
}
```

```rust
use backon::{ExponentialBuilder, Retryable};

pub async fn fetch_json<T: DeserializeOwned>(
    client: &Client,
    url: &str,
    params: &[(&str, &str)],
    sem: &Semaphore,
) -> Result<T, FetchError> {
    let _permit = sem.acquire().await?;
    
    let body = (|| async {
        let resp = client.get(url).query(params).send().await?;
        resp.error_for_status()?.json().await
    })
    .retry(&ExponentialBuilder::default()
        .with_min_delay(Duration::from_secs(1))
        .with_max_delay(Duration::from_secs(10))
        .with_max_times(3))
    .when(|e| matches!(e, FetchError::Timeout(_) | FetchError::Status(_)))
    .await?;
    
    Ok(serde_json::from_slice(&body)?)
}
```

## 3. FastestBilibiliDownloader 复评

> 实际是 Go（96.3% Go），不是 C#。30 worker select 模式。

**致命缺陷**:
1. **无登录/鉴权** —— 没 cookie/QR/token
2. **使用已废弃的 v2 playurl** —— `interface.bilibili.com/v2/playurl?appkey=...&sign=md5`
3. **appkey 硬编码** + entropy 反解 —— `rbMCKn@KuamXWlPMoJGsKcbiJKUfkPF_8dABscJntvqhRSETg` → 这是 B 站 WEB 端历史 appkey，但接口已停用 → 只能拿到 480P 以下
4. **限流 100μs/tick** —— 10000 req/s 会触发反爬
5. **FFmpeg 合并被注释** —— 多分片 FLV 不会合并
6. **`log.Fatalln` 退出** —— 一处错误全进程死
7. **BV 转换算法正确**（`fZodR9...` 表 + 偏移 + 异或）

**结论**: 这个项目是 **BBDown 简化版的"反例"**——它把"无登录 + 旧接口 + 激进限流 + 缺合并"全占了。**Rust 改写时**:
- **绝对不要**学它的接口选择（用 BBDown 的 4-mode playurl）
- **绝对不要**用 100μs 限流
- **可借鉴**: 30 worker select 模式（用 tokio::select! + channel 模拟）
- **可借鉴**: Bv2av 算法（标准表）

### 4.3 关键模块映射 (BBDown + Yutto → Rust)

| BBDown 组件 | Yutto 对应 | Rust 实现 | 备注 |
|---|---|---|---|
| `BBDown.Core.Config` 全局 static | `FetcherContext` 全局 | `Arc<RwLock<AppState>>` + DI | 解决竞态 |
| `IFetcher` + `FetcherFactory` | `Extractor` ABC (12 个 + 5 batch) | `trait Fetcher` + `enum Fetcher` + match | Rust 枚举匹配比 C# 字典分发更优雅 |
| `BBDownLoginUtil` | `login.py` + `segno` | `bili-login` crate, 5 步状态机 | 用 `tokio::time::interval` 轮询 |
| `Parser.GetPlayJsonAsync` (4 mode) | `api.ugc_video.get_ugc_video_playurl` | `bili-core::api::PlayUrlClient` (enum 4 mode) | **Yutto 用 `fnval=4048` 一次拿全** |
| `Parser.WbiSign` | `utils.wbi` | `bili-core::api::wbi::sign()` | 纯函数，易测 |

### 4.4 反爬关键点 (Rust 实现要特殊处理)

| 反爬机制 | 应对 |
|---|---|
| **WBI 签名** | 启动时拿 img_key/sub_key，缓存 1h（实际官方刷新频率）|
| **buvid3/buvid4/b_nut** | 启动时**主动** `GET https://www.bilibili.com/` 拿完整 cookie 集 |
| **gaia-vtoken** | 见 `nilaoda/gaia-vtoken` 反解（仍在更新）|
| **`__04` 指纹** | 见 `social-` cookie 系列，按 API 要求携带 |
| **风控降级** | 实现 5 级回退：APP → TV → Web-bangumi → Web → HTML5 |
| **限流** | `governor::RateLimiter` 50 req/s + 抖动；登录后 200 req/s |
| **重试** | `backoff::exponential` + 429 时 Respect `Retry-After` |
| **连接复用** | 全局 `reqwest::Client` 单例；不要每个命令新建 |
| **gzip / br** | reqwest 默认开启 |

### 4.8 风险点 (需要决策)

1. **APP gRPC protobuf 是否维护?**
   - 官方 proto 文件可能在 `nilaoda/bilibili-API` 仓库
   - 可能被弃用 → 只能走 TV + Web 拿 1080P

2. **ffmpeg 依赖怎么处理?**
   - 方案 A: `ffmpeg-sidecar` crate 启动外部 ffmpeg（推荐）
   - 方案 B: `ffmpeg-next` 静态链接（编译痛苦, **不推荐**）
   - 方案 C: 调系统 PATH 上的 ffmpeg（最简单）

3. **杜比视界 MP4Box 依赖?**
   - GPAC 项目的 `mp4box` 二进制 ~ 3MB
   - 优先用 ffmpeg 5.0+ (要求用户升级)

4. **GUI 选型?**
   - **终端**: `ratatui`（最轻量, 跨平台）
   - **桌面**: Tauri 2 + 已有 `kingwingfly/bilibili-downloader-rs` 38 stars 可参考
   - **Web**: axum 后端 + 前端独立

5. **WBI 算法是否会变?**
   - 历史上 B 站会微调 `mixin_key` 表
   - 持续从 `nilaoda/BBDown` 同步（即使原仓 archived, fork 还在更新）

## 5. 总结

### BBDown (1) 是 Rust 改写的**协议层最佳蓝本**
- 13.8k stars 验证了功能完整
- 5 种 playurl 接口全部开源了反爬策略
- **唯一缺陷**: 原仓 archived, 需要从 `Jared-02/BBDown` 等活跃 fork 同步更新

### Yutto (2) 是 Rust 改写的**工程化最佳范本** ⭐
- 当前最活跃的 B 站下载器（pushedAt 2026-05-31）
- 12 个 Fetcher + 5 batch 变体（比 BBDown 更细分）
- **`fnval=4048` 在 WEB 端拿全画质 (8K/HDR/DV/AV1) —— 不用走 APP gRPC 也能拿高质量**
- **Rust 弹幕库 `biliass-core` 是最直接的 Rust 改写参考**（用 rayon 并行 + 完整 ASS 渲染）
- 多 profile 认证、pydantic 配置、链式 FFmpeg、协程并发下载都是现代化设计
- ⚠️ **GPL-3.0 传染** —— 参考算法可, 复制代码会让你的项目传染

### FastestBilibiliDownloader (3) 只能当反例
- 它的所有架构选择 (无登录/旧接口/激进限流) 都不应学
- **唯一可借鉴**: 30 worker select 模式 (但 Rust 用 tokio::select 实现更自然)

| 登录 | ✅ WEB+TV | ✅ WEB | ❌ |
| 活跃度 | archived (5/2026) | 持续更新 | 持续更新 |

| 配置 | JSON | TOML + pydantic | 无 |
| License | MIT | **GPL-3.0** | 不详 |

| 维度 | BBDown (C#) / Yutto (Python+Rust) | 纯 Rust 实现 |
|---|---|---|
| 启动 | 500ms (C#) / 200ms (Py) | **10ms** |
| 内存 | 200MB (C#) / 80MB (Py) | **30MB** |
| 单一二进制 | ❌ 需 .NET 8 (C#) / ❌ 需 Python+轮子 (Py) | ✅ **5MB native** |
| 并发性能 | ThreadPool / asyncio | **tokio M:N (10x 吞吐)** |

### 推荐决策
- **模式 A: 重写 1.0 (推荐)**: 按 BBDown + Yutto 融合功能 1:1 复刻
  - 协议层抄 BBDown (清晰), 工程层抄 Yutto (现代)
  - 把 `biliass-core` 用 Rust 重新实现 (避免 GPL 传染)
  - 优先 CLI (`bili-cli`) → HTTP server → ratatui TUI
- **模式 B: 差异化**: 跳过 aria2c 自带的复杂性, 强化 DASH 并发 + 互动式 TUI 选 track + 内置 TUI 弹幕预览
- **模式 C: 集成 kingwingfly/bilibili-downloader-rs**: 在 Tauri 项目基础上, 把 Tauri 1.3 升到 2.0 + 接入全功能 core

**推荐从模式 A 开始**——分阶段把 `bili-core` 跑通即可, GUI 是最后的事。

**关键里程碑调整** (相比 v1 报告):
- M5 "杜比/全质量" 大幅简化：~~APP gRPC + prost 集成 2 周~~ → 1 周（用 `fnval=4048` 走 WEB）
- 新增 M3.5: 弹幕库移植 (从 biliass-core 改写为纯 Rust crate, 1 周)
- M4 (番剧) 复杂度降低：yutto 已验证 WEB API 拿番剧可行

## 附录 C: 协议层抓取顺序（首次完整下载流程）

```
1. GET https://www.bilibili.com/         → 拿 buvid3/buvid4/b_nut
2. GET /x/web-interface/nav              → 检查登录态 + 拿 img_key/sub_url
3. (optional) GET /x/passport-login/web/qrcode/generate + poll  → 登录
4. 解析 URL → Fetcher::fetch(id)         → VInfo (含所有 pages)
5. 对每个 page:
   5.1 GET /x/web-interface/view?bvid=  → 拿 aid, cid
   5.2 GET /x/player/wbi/v2?cid=        → 拿 dash url
        (5.2.1 加 w_rid 签名)
   5.3 if 杜比/8K:
        走 APP gRPC playview 接口
   5.4 下载 video track (m4s 段)        → /range
   5.5 下载 audio track (m4s 段)        → /range
   5.6 下载 cover                        → single
   5.7 下载 danmaku                      → /xml or /pb
   5.8 下载 subtitles                   → 5 套 API
   5.9 ffmpeg 混流                       → mux to mp4
   5.10 ffmpeg 写 metadata               → -metadata
6. 输出最终 .mp4 + .xml/.ass + .srt
```
